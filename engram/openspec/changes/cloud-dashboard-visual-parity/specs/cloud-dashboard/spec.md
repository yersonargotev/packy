# Delta Spec: cloud-dashboard (cloud-dashboard-visual-parity)

## Change metadata

- Change: cloud-dashboard-visual-parity
- Capability: cloud-dashboard
- Kind: ADDITIVE (extends base spec from cloud-dashboard-parity)
- Base version: cloud-dashboard/v1 (archived under cloud-dashboard-parity)

## Scope addition

This delta adds full visual and interaction parity requirements to the `cloud-dashboard` capability. The base spec (`cloud-dashboard-parity`) validated route ownership, auth redirects, and read-model queries but did NOT require real static assets, rich templ component output, or HTMX wiring. This delta specifies: (1) static asset byte-size floors that prove real files are embedded, (2) deterministic templ code generation pinned to `v0.3.1001`, (3) flat row type extensions enabling detail pages, (4) an optional `GetDisplayName` auth bridge on `MountConfig`, (5) project sync control persistence via a new SQL table, (6) a `SystemHealth` aggregator for the admin surface, (7) the full 11-endpoint HTMX surface, (8) structural layout requirements (ribbon, hero, tabs, footer), (9) HTMX attribute presence per endpoint, (10) push-path pause enforcement with loud failure, (11) an insecure-mode regression guard, (12) copy parity strings for every dashboard tab, (13) admin-gated mutation enforcement, and (14) a Principal bridge that maps chunk-centric identity to display name. All requirements are testable via `go test ./...` without a browser.

## ADDED Requirements

### REQ-100: Static asset byte-size floors

**Statement**: The embedded static filesystem MUST serve `htmx.min.js` with a size of at least 40 000 bytes, `pico.min.css` with a size of at least 60 000 bytes, and `styles.css` with a size of at least 20 000 bytes. Stub or placeholder files that are smaller than these floors MUST cause the test to fail loudly.

**Scenarios**:

- **Happy path — real assets embedded**: GIVEN the integrated repo has real static files copied from the legacy `engram-cloud` source, WHEN `fs.Stat(StaticFS, "static/htmx.min.js")` is called, THEN the returned `FileInfo.Size()` is greater than or equal to 40 000 bytes.
- **Happy path — pico and styles**: GIVEN the same embedded FS, WHEN `fs.Stat(StaticFS, "static/pico.min.css")` and `fs.Stat(StaticFS, "static/styles.css")` are called, THEN sizes are at least 60 000 bytes and 20 000 bytes respectively.
- **Error path — stub files present**: GIVEN the static directory contains 42-byte or 46-byte placeholder files, WHEN the byte-size assertions run, THEN the test fails with a clear message showing the actual size and the floor.
- **Edge case — file missing from embed**: GIVEN the `//go:embed static` directive does not capture a file (e.g., due to a rename), WHEN `fs.Stat` is called, THEN an error is returned and the test fails before attempting a size comparison.

### REQ-101: Templ deterministic regeneration and version pin

**Statement**: The files `components_templ.go`, `layout_templ.go`, and `login_templ.go` MUST be checked in to the repository and MUST begin with the comment `// Code generated by templ`. The `go.mod` file MUST declare `github.com/a-h/templ v0.3.1001` as a direct dependency. `components_templ.go` MUST be at least 100 000 bytes to confirm the full component surface is generated. A `tools/tools.go` file with build tag `tools` MUST import `github.com/a-h/templ/cmd/templ` so `go mod tidy` retains the binary dependency.

**Scenarios**:

- **Happy path — generated files present**: GIVEN the apply phase has run `templ generate` and committed the output, WHEN `os.Stat` is called on each `*_templ.go` path relative to the package directory, THEN all three files exist and their first line matches `// Code generated by templ`.
- **Happy path — size floor for components**: GIVEN `components_templ.go` contains the full 30+ component surface, WHEN its size is measured via `os.Stat`, THEN the returned size is at least 100 000 bytes.
- **Error path — stale generated file**: GIVEN a contributor regenerates with a different templ version, WHEN the checked-in file header is read, THEN the test SHOULD detect the version mismatch via the generator comment and signal a warning.
- **Edge case — version pin in go.mod**: GIVEN `go.mod` is read, WHEN the file is searched for `github.com/a-h/templ`, THEN the string `v0.3.1001` is present on the same line.

### REQ-102: Row types carry detail fields

**Statement**: `DashboardObservationRow` MUST include the fields `Content string`, `TopicKey string`, `ToolName string`, and `ChunkID string`. `DashboardSessionRow` MUST include `EndedAt string`, `Summary string`, and `Directory string`. `DashboardPromptRow` MUST include `ChunkID string`. All fields MUST be populated from the chunk JSON payload during `buildDashboardReadModel` processing; they MUST NOT be left zero-valued when the payload contains the corresponding data.

**Scenarios**:

- **Happy path — observation row carries content**: GIVEN a chunk JSON payload with `type = "observation"` and a non-empty `content` field, WHEN `buildDashboardReadModel` processes it, THEN the resulting `DashboardObservationRow.Content` equals the payload `content` value.
- **Happy path — session row carries EndedAt**: GIVEN a chunk JSON payload representing a session close event with a timestamp, WHEN the read model is built, THEN `DashboardSessionRow.EndedAt` reflects that timestamp.
- **Error path — missing payload field**: GIVEN a chunk JSON payload that omits `topic_key`, WHEN the row is built, THEN `DashboardObservationRow.TopicKey` is the empty string and no error is returned.
- **Edge case — ChunkID globally unique per row**: GIVEN two observations in the same session, WHEN the read model is built, THEN each `DashboardObservationRow.ChunkID` is distinct and matches the source chunk identifier.

### REQ-103: MountConfig GetDisplayName optional field and nil fallback

**Statement**: `dashboard.MountConfig` MUST include the field `GetDisplayName func(r *http.Request) string`. When this field is `nil`, every handler that invokes it MUST return the literal string `"OPERATOR"` as the display name. When the field is set, handlers MUST invoke it for every request that requires a user display name, including admin sync toggle `updatedBy` attribution.

**Scenarios**:

- **Happy path — nil fallback renders OPERATOR**: GIVEN `MountConfig.GetDisplayName` is `nil`, WHEN a handler renders a page that shows the current user display name, THEN the rendered HTML contains the string `OPERATOR`.
- **Happy path — custom closure invoked**: GIVEN `MountConfig.GetDisplayName` returns `"alice"` for a specific request, WHEN the dashboard page handler processes that request, THEN the rendered HTML contains `alice` in the user-info section.
- **Error path — nil closure not called**: GIVEN `GetDisplayName` is `nil`, WHEN the handler calls the internal `displayName(r)` helper, THEN no nil-pointer panic occurs and the return value is `"OPERATOR"`.
- **Edge case — admin sync toggle updatedBy attribution**: GIVEN `GetDisplayName` returns `"bob"`, WHEN `POST /dashboard/admin/projects/{name}/sync` is processed, THEN `SetProjectSyncEnabled` is called with `updatedBy = "bob"`.

### REQ-104: Project sync control persistence

**Statement**: `CloudStore` MUST expose the methods `IsProjectSyncEnabled(project string) (bool, error)`, `SetProjectSyncEnabled(project string, enabled bool, updatedBy, reason string) error`, `GetProjectSyncControl(project string) (*ProjectSyncControl, error)`, and `ListProjectSyncControls() ([]ProjectSyncControl, error)`. These methods MUST be backed by a SQL table `cloud_project_controls` with the schema: `project TEXT PRIMARY KEY, sync_enabled BOOLEAN NOT NULL DEFAULT 1, paused_reason TEXT, updated_at TEXT NOT NULL, updated_by TEXT`. The table MUST be created via `CREATE TABLE IF NOT EXISTS` during `CloudStore` initialization so the migration is idempotent.

**Scenarios**:

- **Happy path — enable/disable round-trip**: GIVEN a project that has never been configured, WHEN `SetProjectSyncEnabled("proj", false, "operator", "maintenance")` is called, THEN `IsProjectSyncEnabled("proj")` returns `false, nil` and `GetProjectSyncControl("proj")` returns a record with `SyncEnabled = false` and `PausedReason = "maintenance"`.
- **Happy path — list returns all controls**: GIVEN two projects have been configured with sync enabled and disabled, WHEN `ListProjectSyncControls()` is called, THEN the returned slice contains both records with correct `SyncEnabled` values.
- **Error path — unknown project defaults to enabled**: GIVEN a project with no row in `cloud_project_controls`, WHEN `IsProjectSyncEnabled("unknown")` is called, THEN the method returns `true, nil` (absent row means sync is allowed).
- **Edge case — idempotent migration**: GIVEN `CloudStore.New()` is called twice on the same SQLite file, WHEN the second call runs the migration, THEN no error is returned and the existing data is preserved.

### REQ-105: Dashboard system health aggregator

**Statement**: `CloudStore` MUST expose `SystemHealth() (DashboardSystemHealth, error)`. The returned `DashboardSystemHealth` struct MUST carry: `DBConnected bool`, `Projects int`, `Contributors int`, `Sessions int`, `Observations int`, `Prompts int`, `Chunks int`. All counts MUST be sourced from the in-memory `dashboardReadModel` (no extra SQL queries). `DBConnected` MUST be `true` when the underlying SQLite connection responds to a ping-equivalent check and `false` when the connection is closed or unavailable.

**Scenarios**:

- **Happy path — health from seeded chunks**: GIVEN the cloudstore has been seeded with 2 projects, 3 sessions, 10 observations, and 5 prompts, WHEN `SystemHealth()` is called, THEN `DashboardSystemHealth.Projects = 2`, `Sessions = 3`, `Observations = 10`, `Prompts = 5`, and `DBConnected = true`.
- **Happy path — empty store**: GIVEN no chunks have been synced, WHEN `SystemHealth()` is called, THEN all integer fields are 0 and `DBConnected = true`.
- **Error path — db connection closed**: GIVEN the SQLite database is closed before calling `SystemHealth()`, WHEN the call is made, THEN `DBConnected = false` and the error field propagates the underlying connection error.
- **Edge case — contributors counted correctly**: GIVEN two chunks share the same `created_by` contributor, WHEN `SystemHealth()` is called, THEN `Contributors = 1` (distinct contributor count, not chunk count).

### REQ-106: Full HTMX endpoint surface (11 endpoints)

**Statement**: The following 11 endpoints MUST be registered and return HTTP 200 with `Content-Type: text/html; charset=utf-8` for an authenticated Principal, HTTP 302 redirect to `/dashboard/login` for an unauthenticated request, and HTTP 403 for non-admin principals on admin mutation routes. The endpoints are:

1. `GET /dashboard/projects/list` — projects list partial
2. `GET /dashboard/projects/{name}/observations` — project observations partial
3. `GET /dashboard/projects/{name}/sessions` — project sessions partial
4. `GET /dashboard/projects/{name}/prompts` — project prompts partial
5. `GET /dashboard/admin/users` — admin users page
6. `GET /dashboard/admin/health` — admin health page
7. `POST /dashboard/admin/projects/{name}/sync` — sync toggle mutation (admin-gated)
8. `GET /dashboard/admin/projects/{name}/sync/form` — sync form partial
9. `GET /dashboard/sessions/{project}/{sessionID}` — session detail page
10. `GET /dashboard/observations/{project}/{sessionID}/{chunkID}` — observation detail page
11. `GET /dashboard/prompts/{project}/{sessionID}/{chunkID}` — prompt detail page

**Scenarios**:

- **Happy path — all GET endpoints return 200 for authed Principal**: GIVEN a valid dashboard session cookie is present, WHEN each of the 11 endpoints is requested, THEN the response status is 200 and the `Content-Type` header starts with `text/html`.
- **Unauthed path — all endpoints redirect to login**: GIVEN no session cookie is present and `RequireSession` returns an error, WHEN any of the 11 endpoints is requested, THEN the response status is 302 and the `Location` header contains `/dashboard/login`.
- **Error path — admin mutation blocked for non-admin**: GIVEN a valid session cookie is present but `IsAdmin(r)` returns `false`, WHEN `POST /dashboard/admin/projects/{name}/sync` is requested, THEN the response status is 403 Forbidden.
- **Edge case — partial endpoints return standalone HTML**: GIVEN an authed HTMX request to `GET /dashboard/projects/list`, WHEN the response body is read, THEN it contains meaningful HTML (at minimum a non-empty `<div>` or `<table>` element) without a full page shell.

### REQ-107: Layout structural parity (ribbon, hero, tabs, footer)

**Statement**: Every full-page dashboard response MUST render an HTML structure containing all of the following CSS class markers and content strings: `shell-body`, `shell-backdrop`, `app-shell`, `shell-header`, `brand-stack`, `shell-nav`, `shell-main`, `shell-footer`. The `status-ribbon` element MUST contain the text `CLOUD ACTIVE`. The hero MUST render `An elephant never forgets.` as the brand subtitle. The footer MUST render `ENGRAM CLOUD / SHARED MEMORY INDEX / LIVE SYNC READY`. The nav MUST include links for all five tabs: DASHBOARD (`/dashboard/`), BROWSER (`/dashboard/browser`), PROJECTS (`/dashboard/projects`), CONTRIBUTORS (`/dashboard/contributors`), and ADMIN (`/dashboard/admin`).

**Scenarios**:

- **Happy path — shell structure present on home page**: GIVEN an authed GET to `/dashboard/`, WHEN the response body is parsed, THEN all eight CSS class markers are found in the HTML.
- **Happy path — ribbon and footer text present**: GIVEN an authed GET to `/dashboard/`, WHEN the body is scanned, THEN the strings `CLOUD ACTIVE` and `ENGRAM CLOUD / SHARED MEMORY INDEX / LIVE SYNC READY` are present.
- **Happy path — hero brand copy present**: GIVEN an authed GET to any full-page dashboard route, WHEN the body is scanned, THEN `An elephant never forgets.` is present.
- **Edge case — admin tab hidden for non-admin**: GIVEN `IsAdmin(r)` returns `false`, WHEN a full-page route is requested, THEN the rendered HTML does NOT contain an admin nav link (`href="/dashboard/admin"`).

### REQ-108: HTMX attribute presence per endpoint

**Statement**: The HTML rendered by full-page dashboard handlers MUST include `hx-get`, `hx-post`, `hx-target`, `hx-trigger`, and/or `hx-swap` attributes as required by the following mapping. The dashboard home page MUST emit `hx-get="/dashboard/stats"` and `hx-get="/dashboard/activity"` with `hx-trigger="load"`. The browser page MUST emit `hx-get` attributes targeting `/dashboard/browser/observations`, `/dashboard/browser/sessions`, and `/dashboard/browser/prompts` with `hx-target="#browser-content"` and `hx-include="#browser-project"`. The projects page MUST emit `hx-get="/dashboard/projects/list"` with `hx-target="#projects-content"` and `hx-trigger` containing `keyup`. The admin page MUST emit `hx-post` on the sync toggle form pointing to `/dashboard/admin/projects/{name}/sync`.

**Scenarios**:

- **Happy path — home page HTMX wiring**: GIVEN an authed GET to `/dashboard/`, WHEN the body is scanned, THEN the strings `hx-get="/dashboard/stats"`, `hx-get="/dashboard/activity"`, and `hx-trigger="load"` are all present.
- **Happy path — browser page HTMX wiring**: GIVEN an authed GET to `/dashboard/browser`, WHEN the body is scanned, THEN `hx-get="/dashboard/browser/observations"`, `hx-target="#browser-content"`, and `hx-include="#browser-project"` are present.
- **Happy path — projects page HTMX wiring**: GIVEN an authed GET to `/dashboard/projects`, WHEN the body is scanned, THEN `hx-get="/dashboard/projects/list"`, `hx-target="#projects-content"`, and a `hx-trigger` value containing `keyup` are present.
- **Edge case — admin sync toggle form HTMX wiring**: GIVEN an authed admin GET to `/dashboard/admin`, WHEN the body is scanned, THEN at least one `hx-post` attribute targeting a `/dashboard/admin/projects/` path is present for each listed project.

### REQ-109: Push-path pause enforcement with reason code

**Statement**: The push-path handler MUST call `IsProjectSyncEnabled(project)` before accepting any sync mutation for that project. When `IsProjectSyncEnabled` returns `false`, the handler MUST respond with HTTP 409 Conflict and a JSON or text body containing the reason code `sync-paused`. The push path MUST NOT silently drop the mutation or return HTTP 200 when sync is paused.

**Scenarios**:

- **Happy path — push accepted when sync enabled**: GIVEN a project with `sync_enabled = true`, WHEN a push mutation arrives for that project, THEN the handler proceeds normally and returns HTTP 200.
- **Error path — push rejected when sync paused**: GIVEN a project with `sync_enabled = false`, WHEN a push mutation arrives for that project, THEN the handler returns HTTP 409 Conflict and the response body contains the string `sync-paused`.
- **Error path — reason code survives JSON decode**: GIVEN the push client parses the 409 response, WHEN the body is decoded, THEN the `reason` or equivalent field contains `sync-paused` exactly.
- **Edge case — unknown project not paused**: GIVEN a project with no row in `cloud_project_controls`, WHEN a push arrives for that project, THEN `IsProjectSyncEnabled` returns `true` and the push proceeds.

### REQ-110: Insecure-mode login regression guard

**Statement**: When `CloudServer` is constructed with `auth == nil` (insecure/no-auth mode), `GET /dashboard/login` MUST respond with HTTP 303 redirect to `/dashboard/` and MUST NOT render the login form, return HTTP 200, or panic. This behavior MUST be covered by a regression test in `cloudserver_test.go` that constructs a `CloudServer` with a nil authenticator.

**Scenarios**:

- **Happy path — login redirects immediately in insecure mode**: GIVEN `CloudServer` is created with `auth == nil`, WHEN `GET /dashboard/login` is requested, THEN the response status is 303 and the `Location` header is `/dashboard/`.
- **Happy path — dashboard accessible without cookie**: GIVEN `auth == nil`, WHEN `GET /dashboard/` is requested without any session cookie, THEN the response status is 200 (RequireSession passes immediately).
- **Error path — nil auth does not panic on type assertion**: GIVEN `auth == nil`, WHEN any code path that calls `s.auth.(dashboardSessionCodec)` is reached, THEN no nil-pointer panic occurs because `createSessionCookie` is nil and the code path is never entered.
- **Edge case — insecure mode login POST with empty token is blocked**: GIVEN `auth == nil`, WHEN `POST /dashboard/login` is submitted with an empty `token` field, THEN the handler returns the login page with an error message (empty token is still rejected at the application layer).

### REQ-111: Copy parity strings rendered in HTML

**Statement**: The rendered HTML for each full-page dashboard route MUST contain the tab-specific kicker string that matches the legacy `engram-cloud` dashboard. Required strings per route: `/dashboard/browser` MUST contain `KNOWLEDGE BROWSER`; `/dashboard/projects` MUST contain `PROJECT ATLAS`; `/dashboard/contributors` MUST contain `CONTRIBUTOR SIGNAL`; `/dashboard/admin` MUST contain `ADMIN SURFACE`. The login page MUST contain `Engram Cloud` and `CLOUD ACTIVE`. The full-page shell MUST render `Engram Cloud` as the brand name in the header.

**Scenarios**:

- **Happy path — browser kicker**: GIVEN an authed GET to `/dashboard/browser`, WHEN the body is scanned, THEN the string `KNOWLEDGE BROWSER` is present.
- **Happy path — admin kicker**: GIVEN an authed admin GET to `/dashboard/admin`, WHEN the body is scanned, THEN `ADMIN SURFACE` is present.
- **Happy path — login copy**: GIVEN GET to `/dashboard/login`, WHEN the body is scanned, THEN `Engram Cloud` and `CLOUD ACTIVE` are both present.
- **Edge case — projects and contributors kickers**: GIVEN authed GETs to `/dashboard/projects` and `/dashboard/contributors`, WHEN bodies are scanned, THEN `PROJECT ATLAS` and `CONTRIBUTOR SIGNAL` are present respectively.

### REQ-112: Admin gate on toggle mutations

**Statement**: The `POST /dashboard/admin/projects/{name}/sync` handler MUST check `h.cfg.IsAdmin(r)` at handler entry. If `IsAdmin` returns `false`, the handler MUST return HTTP 403 Forbidden immediately and MUST NOT invoke `SetProjectSyncEnabled`. If `IsAdmin` returns `true`, the handler MUST parse the `enabled` and `reason` fields from the POST form, call `SetProjectSyncEnabled`, and respond with HTTP 303 redirect to `/dashboard/admin/projects` with an `HX-Redirect` header when the request is HTMX-driven.

**Scenarios**:

- **Happy path — admin toggle succeeds**: GIVEN `IsAdmin(r)` returns `true` and a valid POST body with `enabled=false` and `reason=maintenance`, WHEN `POST /dashboard/admin/projects/myproject/sync` is called, THEN `SetProjectSyncEnabled("myproject", false, displayName, "maintenance")` is called and the response is 303.
- **Error path — non-admin gets 403**: GIVEN `IsAdmin(r)` returns `false`, WHEN the POST is made, THEN the response is 403 Forbidden and `SetProjectSyncEnabled` is never called.
- **Error path — store error returns 500**: GIVEN `SetProjectSyncEnabled` returns an error, WHEN the POST is processed, THEN the handler returns HTTP 500 and the error is logged.
- **Edge case — HTMX-driven post includes HX-Redirect header**: GIVEN the request has the `HX-Request: true` header and `IsAdmin` is `true`, WHEN the POST succeeds, THEN the response includes the `HX-Redirect` header pointing to `/dashboard/admin/projects`.

### REQ-113: Principal bridge for legacy User reads

**Statement**: The integrated dashboard MUST NOT use `getUsernameFromContext`, `getUserIDFromContext`, or `getEmailFromContext` pattern. All display name lookups MUST go through `h.cfg.GetDisplayName(r)` via the internal `displayName(r *http.Request) string` helper method on `handlers`. All per-user data scoping that the legacy dashboard did via `userID` MUST instead use the project scoping provided by `dashboardAllowedScopes`. No handler MAY reach into the `http.Request` context directly for identity claims.

**Scenarios**:

- **Happy path — display name surfaced through helper**: GIVEN `MountConfig.GetDisplayName` returns a non-empty string, WHEN any full-page handler renders, THEN the display name in the output matches the closure return value.
- **Happy path — all data queries use project scope**: GIVEN `dashboardAllowedScopes` contains `["proj-a", "proj-b"]`, WHEN the browser page is requested, THEN only observations belonging to `proj-a` or `proj-b` appear in results, regardless of who is logged in.
- **Error path — no legacy context injection**: GIVEN the request context does NOT contain username/userID/email values, WHEN any handler runs, THEN no panic or "key not found" error occurs.
- **Edge case — displayName helper never returns empty string**: GIVEN `GetDisplayName` returns an empty string, WHEN `displayName(r)` is called, THEN the helper returns `"OPERATOR"` (empty string is treated as absent).
